# Gym Tracker - v0.2.0 (Plan Detallado)

## 🚀 Objetivos de la iteración
Esta versión tiene como foco terminar los módulos principales, establecer una arquitectura de frontend escalable y empezar a darle forma de producto real.

---

## 🏛️ Arquitectura General
Se adoptará una arquitectura de frontend basada en dominios, separando la lógica en las siguientes capas dentro de cada módulo:

-   `modules/[dominio]/`: Contiene toda la lógica de una funcionalidad.
    -   `features/`: Componentes "Smart" que gestionan estado y datos. Suelen corresponder a una página o una sección compleja.
    -   `components/`: Componentes "Dumb", reutilizables y centrados en la UI. Reciben datos solo por props.
    -   `actions/`: Server Actions de Next.js para las operaciones de backend.
    -   `contexts/`: Contextos de React para el estado local del módulo.
    -   `modules/`: Sub-dominios en caso de ser necesarios.

## 📌 Funcionalidades y Soluciones Técnicas

### 1. Módulo de Log Workout (Refactor)
**Objetivo:** Refactorizar el módulo para alinearla con la nueva arquitectura y mejorar la experiencia de la sesión.

-   **Estructura de Archivos:**
    ```
    src/modules/log-workout/
    └── modules/
        └── session/
            ├── features/
            │   └── workout-session.feature.tsx
            ├── components/
            │   ├── WorkoutExerciseItem.tsx
            │   ├── WorkoutExerciseCompactItem.tsx
            │   └── AddExerciseToWorkout.tsx
            ├── contexts/
            │   └── WorkoutSessionContext/
            └── hooks/
    ```
-   **Plan de Acción:**
    -   La lógica principal de la página se moverá a `workout-session.feature.tsx`.
    -   Los componentes de UI se convertirán en "dumb components" dentro de `components/`.
    -   Se creará un nuevo feature `workout-history.feature.tsx` para visualizar sesiones pasadas.

### 2. Módulo de Rutinas (MVP)
**Objetivo:** Permitir a los usuarios crear, gestionar y utilizar rutinas de entrenamiento.

-   **Modelo de Datos:** Se añadirán los modelos `Routine` y `RoutineExercise` al `schema.prisma`.
-   **Estructura de Archivos:**
    ```
    src/modules/routines/
    ├── features/
    │   ├── routine-list.feature.tsx
    │   └── routine-editor.feature.tsx
    ├── components/
    │   ├── RoutineCard.tsx
    │   └── ExercisePicker.tsx
    └── actions/
        └── routines.actions.ts
    ```
-   **Plan de Acción:**
    -   `/routines`: Mostrará la lista de rutinas usando `routine-list.feature.tsx`.
    -   `/routines/[id]`: Permitirá crear/editar una rutina con `routine-editor.feature.tsx`.

### 3. Módulo de Estadísticas (MVP)
**Objetivo:** Proveer al usuario con métricas visuales sobre su progreso.

-   **Estructura de Archivos:**
    ```
    src/modules/statistics/
    ├── features/
    │   └── statistics-dashboard.feature.tsx
    ├── components/
    │   ├── ProgressChart.tsx
    │   ├── VolumeHistoryChart.tsx
    │   └── ConsistencyCalendar.tsx
    └── actions/
        └── statistics.actions.ts
    ```
-   **Plan de Acción:**
    -   La lógica de cálculo de métricas (volumen, 1RM estimado, etc.) se realizará en el backend a través de `statistics.actions.ts`.
    -   El `statistics-dashboard.feature.tsx` consumirá estas actions y pasará los datos a los componentes de gráficos.

### 4. Sesión de Entrenamiento Activa
**Objetivo:** Permitir que una sesión de entrenamiento persista aunque el usuario navegue a otras secciones de la app. **(Se implementará tras los refactors)**.

-   **Solución Técnica:**
    1.  **Base de Datos:** Se creará una tabla `WorkoutSession` en `schema.prisma`. Al iniciar un entrenamiento, se crea un registro con `status: 'IN_PROGRESS'`.
    2.  **LocalStorage:** Se guardará únicamente el `id` de la sesión activa en el navegador (`activeWorkoutSessionId`).
    3.  **Contexto Global:** Un `ActiveSessionProvider` envolverá la aplicación. Al cargar, revisará `localStorage`, y si hay un ID, recuperará la sesión de la BD mediante una server action, poniéndola a disposición de toda la app.

### 5. Sidebar y Perfil
**Objetivo:** Reemplazar el botón genérico de configuración por un acceso al perfil de usuario.

-   **Plan de Acción:** El enlace del sidebar apuntará a la nueva página `/settings`.

### 6. Configuración v0
**Objetivo:** Crear una página básica de configuración de usuario.

-   **Página `/settings`:**
    -   **Perfil de Usuario:** Formulario (`ProfileForm.tsx`) para editar nombre y foto de perfil, llamando a una server action `updateUserProfile`.
    -   **Apariencia:** Switch (`AppearanceSettings.tsx`) para gestionar el Dark Mode a través de `next-themes`.
