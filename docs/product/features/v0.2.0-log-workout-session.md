## 1. Contexto y problema

### El problema crítico

El usuario principal —el "Entrenador Consciente"— no puede usar la aplicación para su propósito principal: registrar entrenamientos en el gimnasio. La razón es la **pérdida catastrófica de progreso** cuando el navegador se minimiza.

En el contexto real de uso, el usuario necesita:

- Cambiar la música entre series
- Responder mensajes urgentes
- Consultar un video de técnica
- Ver la hora

Cada vez que el navegador deja de estar en primer plano, la sesión activa se pierde. Esto hace que la aplicación sea **inútil para su propósito principal**.

Adicionalmente, el usuario necesita poder **editar y corregir datos libremente** durante la sesión sin que cada cambio se guarde permanentemente. Necesita un espacio de "borrador" donde pueda actualizar datos con retraso, corregir errores, y modificar series ya registradas antes de hacer el commit final.

### ¿Qué pasa si NO lo construimos?

Sin persistencia de sesión:

- El usuario abandona la aplicación y vuelve a Notion, Excel o la libreta física
- Todo el trabajo invertido en rutinas y catálogo de ejercicios se desperdicia
- El producto no alcanza product-market fit

---

## 2. Objetivos

1. **Persistencia de sesión garantizada (local)**: Una sesión de entrenamiento activa debe sobrevivir a minimizaciones del navegador, navegación a otras secciones de la app, y cierres accidentales. El estado del borrador se mantiene en el cliente (localStorage) durante toda la sesión.

2. **Claridad durante la sesión**: El usuario debe saber de un vistazo:
   - En qué ejercicio y serie se encuentra actualmente
   - Cuál es el siguiente ejercicio/serie
   - Cuánto peso debe usar en la siguiente serie (basado en sesiones anteriores)
   - Cuántas series totales tiene el ejercicio actual

3. **Registro sin fricción**: La carga de datos (peso, reps, RPE) debe ser posible con una sola mano, sin necesidad de usar el teclado numérico del celular.

4. **Edición libre durante la sesión**: El usuario debe poder modificar cualquier dato ingresado (peso, reps, RPE) en cualquier momento durante la sesión, sin restricciones. No hay commit parcial a la base de datos hasta el final.

---

## 3. Non-goals

Los siguientes puntos están **explícitamente fuera del alcance** de esta versión:

- **Temporizador de descanso automático entre series**: El usuario puede usar el temporizador existente manualmente, pero no hay integración automática al completar una serie.

- **Sincronización en tiempo real entre dispositivos**: La sesión persiste en el dispositivo local y en la base de datos, pero no hay sincronización activa entre múltiples dispositivos simultáneamente.

- **Modo offline completo**: Aunque la sesión se persiste localmente, se asume conectividad para el inicio de la sesión y sincronización con el backend.

- **Integración con wearables** (Apple Watch, Garmin, etc.): No se incluyen notificaciones ni monitoreo desde dispositivos externos.

- **Análisis avanzado de rendimiento en tiempo real**: No se muestran gráficos ni análisis durante la sesión activa (esto pertenece al módulo de estadísticas post-sesión).

---

## 4. Usuarios afectados

**El "Entrenador Consciente"** — usuario que sigue un programa de fuerza estructurado (Push/Pull/Legs, Upper/Lower, 5/3/1, etc.), autorregula sus cargas según cómo se siente cada día, y necesita precisión sin fricción mientras entrena.

**Contexto de uso principal**:

- **Durante el entrenamiento**: En el gimnasio, entre series, con las manos posiblemente sudadas. Necesita registrar datos rápidamente sin perder el ritmo.
- **Situaciones de interrupción**: Minimiza el navegador para cambiar música, responder mensajes, o consultar algo. Espera que su progreso esté intacto al volver.

---

## 5. User stories

- Como entrenador consciente, quiero que mi sesión de entrenamiento persista aunque minimice el navegador, para no perder el progreso de mi entrenamiento actual.

- Como entrenador consciente, quiero ver claramente en qué ejercicio y serie estoy actualmente, para no perder el contexto durante la sesión.

- Como entrenador consciente, quiero saber cuál es el siguiente ejercicio/serie, para prepararme mentalmente antes de terminar la serie actual.

- Como entrenador consciente, quiero ver cuánto peso usé en sesiones anteriores del mismo ejercicio, para poder autorregular mi carga sin tener que buscar en el historial.

- Como entrenador consciente, quiero registrar peso, repeticiones y RPE con controles táctiles (sin teclado), para poder hacerlo rápidamente con una mano entre series.

- Como entrenador consciente, quiero poder cambiar o agregar ejercicios durante la sesión si una máquina está ocupada, sin perder el progreso de las series ya completadas.

- Como entrenador consciente, quiero poder navegar a otras secciones de la app (ver ejercicios, rutinas, etc.) y que mi sesión siga activa, para poder consultar información sin abandonar mi entrenamiento.

- Como entrenador consciente, quiero poder editar o eliminar una serie que ya marqué como completada, para corregir errores de tipeo sin necesidad de empezar de nuevo.

- Como entrenador consciente, quiero que todos mis datos se guarden en la base de datos solo cuando yo decida finalizar el entrenamiento, para tener control total sobre qué se registra permanentemente.

## 6. Flujo principal

### Iniciar una sesión de entrenamiento

1. El usuario accede a la sección "Entrenar" desde la navegación principal.
2. Ve las opciones: "Entrenamiento libre", "Continuar sesión" (si hay una activa), o seleccionar una rutina y día específicos.
3. Selecciona una opción y el sistema **crea un borrador de sesión en el cliente** (localStorage).
   - No se crea ningún registro en la base de datos en este momento.
   - El estado completo de la sesión (ejercicios, series planificadas) se almacena en localStorage.

### Durante la sesión — Modo Borrador

4. El usuario ve el ejercicio actual con:
   - Indicador de progreso (serie X de Y)
   - Peso usado en la última sesión (si existe historial previo en BD)
   - Controles para ingresar peso actual
   - Controles para ingresar repeticiones
   - Opcional: selector de RPE

5. El usuario completa los campos y toca "Marcar serie como completada".
   - Los datos se guardan **únicamente en localStorage** (estado del borrador).
   - **No hay escritura en la base de datos**.

6. En cualquier momento durante la sesión, el usuario puede:
   - **Editar cualquier serie ya completada**: cambiar peso, reps o RPE.
   - **Deshacer el último cambio**: restaurar el estado anterior de una serie.
   - **Eliminar una serie completada**: removerla del borrador.
   - **Agregar series extras**: más allá de las planificadas inicialmente.

7. Si hay más series del mismo ejercicio, se muestra la siguiente. Si no, se muestra el siguiente ejercicio o la opción de agregar más.

### Navegación y persistencia local

8. En cualquier momento, el usuario puede:
   - **Minimizar el navegador**: el borrador persiste en localStorage. Al volver, recupera exactamente donde estaba.
   - **Navegar a otra sección de la app**: el indicador de "Sesión en progreso" permanece visible. Los datos del borrador no se pierden.
   - **Cerrar accidentalmente el navegador**: al volver a abrir la app dentro de 24 horas, se ofrece "Continuar sesión" recuperando el borrador de localStorage.

### Finalizar el entrenamiento — Commit único

9. Al completar todos los ejercicios (o cuando el usuario decide terminar), toca "Finalizar entrenamiento".

10. El sistema realiza el **commit masivo a la base de datos**:
    - Crea el registro `WorkoutSession` en BD con estado `COMPLETED`.
    - Crea todos los registros `WorkoutExercise` asociados.
    - Crea todos los registros `SetEntry` con los datos finales del borrador.
    - Todo en una única transacción atómica.

11. Si el commit es exitoso:
    - Se limpia el localStorage.
    - Se muestra un resumen de la sesión completada.
    - Los datos ahora están disponibles en el historial.

12. Si ocurre un error durante el commit:
    - Se muestra un mensaje de error al usuario.
    - El borrador se mantiene en localStorage (no se pierden datos).
    - Se permite reintentar el guardado.

### Cancelar sesión

13. El usuario puede cancelar la sesión en cualquier momento:
    - Se muestra confirmación: "¿Descartar todos los cambios?".
    - Si confirma: se limpia localStorage, no se crea ningún registro en BD.
    - Si hay datos previos en BD de sesiones pasadas, estos no se ven afectados.

### Interrupciones y cambios durante el borrador

14. Si una máquina está ocupada, el usuario puede:
    - Reordenar ejercicios pendientes (mover el actual al final, por ejemplo).
    - Agregar un nuevo ejercicio "on-the-fly" desde el catálogo o creándolo nuevo.
    - Saltar un ejercicio (marcarlo como skipped) — se guarda en el borrador y se incluye en el commit final.
    - Eliminar un ejercicio agregado por error (si no tiene series completadas).

## 7. Criterios de aceptación

### Persistencia local (Modo Borrador)

- [ ] Al iniciar una sesión, se crea un **borrador en localStorage** con el estado completo de la sesión (ejercicios, series planificadas). **No se crea registro en BD**.
- [ ] Todos los cambios durante la sesión (peso, reps, RPE, series completadas) se guardan **únicamente en localStorage**.
- [ ] Si el usuario minimiza el navegador y vuelve, el borrador se recupera automáticamente desde localStorage con todos los datos intactos.
- [ ] Si el usuario navega a otra sección de la app, el indicador de "Sesión en progreso" permanece visible y el borrador sigue disponible.
- [ ] Si el usuario cierra el navegador y vuelve a abrir la app dentro de las 24 horas, se ofrece "Continuar sesión" recuperando el borrador de localStorage.
- [ ] Al cancelar una sesión, se limpia el localStorage sin crear ningún registro en BD.

### Claridad durante la sesión

- [ ] El usuario puede ver claramente en qué ejercicio está (nombre destacado, posición en la lista).
- [ ] Se muestra el indicador de progreso del ejercicio actual: "Serie X de Y".
- [ ] Se muestra el peso utilizado en la última sesión del mismo ejercicio (si existe historial).
- [ ] El usuario puede ver cuál es el siguiente ejercicio en la lista (vista previa).
- [ ] El indicador de sesión activa es visible en la navegación principal mientras haya una sesión en curso.

### Registro sin fricción

- [ ] Los campos de peso y repeticiones usan controles stepper táctiles (no requieren teclado).
- [ ] El área táctil de los controles es de al menos 44x44px.
- [ ] El usuario puede completar una serie con una sola mano en menos de 5 segundos.
- [ ] Los cambios se guardan automáticamente en el borrador de localStorage (no requieren botón "Guardar" explícito durante la sesión).

### Flexibilidad durante la sesión (Modo Borrador)

- [ ] El usuario puede agregar ejercicios a la sesión en curso desde el catálogo existente (solo en el borrador, no en BD).
- [ ] El usuario puede crear un ejercicio nuevo "on-the-fly" desde la sesión activa.
- [ ] El usuario puede reordenar los ejercicios pendientes (mover ejercicios hacia arriba/abajo en la lista).
- [ ] El usuario puede saltar un ejercicio (marcarlo como skipped) con opción de indicar razón.
- [ ] **El usuario puede editar cualquier serie ya completada**: cambiar peso, reps o RPE en cualquier momento antes de finalizar.
- [ ] **El usuario puede eliminar una serie completada**: removerla del borrador si fue ingresada por error.
- [ ] **El usuario puede deshacer el último cambio**: restaurar el estado anterior de una serie (al menos 1 nivel de undo).
- [ ] Las series completadas de un ejercicio no se pierden si el ejercicio se reordena o se salta.

### Guardado final (Commit a BD)

- [ ] Al tocar "Finalizar entrenamiento", todos los datos del borrador se guardan en la base de datos en una única operación atómica.
- [ ] El commit incluye: sesión, ejercicios, y todas las series con sus datos finales.
- [ ] No se crea ningún registro parcial en BD si el commit falla (transacción atómica).
- [ ] Si el commit es exitoso, se limpia localStorage y se muestra confirmación al usuario.
- [ ] Si el commit falla, se muestra error al usuario, se mantiene el borrador en localStorage, y se permite reintentar.
- [ ] Una vez finalizada la sesión (commit exitoso), los datos se muestran en el historial y se usan para calcular pesos en futuras sesiones.

## 8. Datos y entidades involucradas

### Durante la sesión (Modo Borrador - Solo Cliente)

- **El estado completo de la sesión se almacena en `localStorage`** mientras está activa. Esto incluye:
  - Metadata de la sesión (nombre, fecha de inicio, rutina/día de origen si aplica).
  - Lista de ejercicios con su orden.
  - Series planificadas y completadas con peso, reps, RPE.
  - Estado de ejercicios saltados (skipped) y sus razones.
  - Timestamp de última modificación (para detectar sesiones "abandonadas").

- **No se crea ningún registro en la base de datos** durante esta fase.
- **Los datos de peso usado en sesiones anteriores** se consultan de la BD al inicio de la sesión para mostrar referencias, pero no se escribe nada nuevo.

### Al finalizar (Commit a Base de Datos)

- **Transacción atómica única**: Todo el borrador se persiste en BD de una sola vez.
- **Entidades creadas**:
  - `WorkoutSession`: Registro principal con estado `COMPLETED`, fecha/hora de inicio y fin, duración total.
  - `WorkoutExercise`: Uno por ejercicio realizado, con referencia al ejercicio del catálogo y orden en la sesión.
  - `SetEntry`: Una por serie completada, con peso, repeticiones, RPE opcional, timestamp, y orden dentro del ejercicio.
- **Si el commit falla**, no se crea ningún registro parcial (rollback automático), y el borrador permanece en localStorage para reintento.

### Post-sesión

- Una vez completado el commit, la sesión está disponible en el **historial del usuario**.
- Los datos de esta sesión se usarán como referencia ("último peso usado") en futuras sesiones del mismo ejercicio.

## 9. Fases de implementación

### Fase 1 — MVP (Persistencia Local y Commit Único)

Alcance mínimo que entrega valor inmediato:

- [ ] Implementar persistencia de sesión en modo borrador:
  - Guardar estado completo de sesión en `localStorage` (no solo ID)
  - Recuperar borrador automáticamente al volver a la app
  - Sincronizar estado entre pestañas
  - **NO guardar en BD hasta el final**

- [ ] Implementar commit único al finalizar:
  - Server action que reciba todo el estado del borrador
  - Transacción atómica: crear sesión, ejercicios y series en una operación
  - Manejo de errores con reintentos y preservación del borrador

- [ ] Crear indicador de sesión activa en la navegación principal (visible desde cualquier sección de la app).

- [ ] Mejorar claridad de la sesión activa:
  - Destacar visualmente el ejercicio actual
  - Mostrar indicador "Serie X de Y" claramente visible
  - Mostrar peso de la última sesión del mismo ejercicio
  - Vista previa del siguiente ejercicio

- [ ] Controles táctiles optimizados:
  - Steppers de peso y reps con área táctil mínima 44x44px
  - Guardado automático en localStorage (sin botón explícito durante sesión)

- [ ] Funcionalidades de edición del borrador:
  - Editar serie completada (cambiar peso/reps/RPE)
  - Eliminar serie completada
  - Undo de último cambio (al menos 1 nivel)

### Fase 2 — Iteración (Flexibilidad)

Funcionalidades que mejoran la experiencia pero no bloquean el uso:

- [ ] Agregar ejercicios a sesión en curso (desde catálogo o creación on-the-fly).
- [ ] Reordenar ejercicios pendientes dentro de la sesión.
- [ ] Saltar ejercicios con opción de indicar razón.
- [ ] Resumen post-sesión con volumen total, tiempo transcurrido y PRs personales.
- [ ] Acceso al historial de sesiones completadas desde el dashboard.

### Fase 3 — Optimización

Mejoras de performance y polish:

- [ ] Sincronización en segundo plano para sesiones largas.
- [ ] Modo "No molestar" que silencia notificaciones del sistema durante la sesión.
- [ ] Atajos de teclado para usuarios en desktop.
- [ ] Integración con APIs de salud (Apple Health, Google Fit) para sincronizar datos.

## Notas de implementación

### Exclusiones intencionales de este PRD

Este documento se enfoca exclusivamente en el módulo `log-workout`. Las siguientes funcionalidades mencionadas en `docs/upcoming/v0.2.0.md` se tratarán en PRDs separados:

- **Módulo de Estadísticas**: Dashboard de progreso, gráficos de volumen y 1RM estimado.
- **Módulo de Configuración**: Página `/settings` con perfil de usuario y preferencias de tema.
- **Mejoras adicionales de Sidebar**: Navegación reestructurada.

### Dependencias del módulo de rutinas

El módulo `log-workout` depende del módulo `routines` ya implementado:

- Una sesión puede iniciarse desde una rutina y día específicos
- La estructura de ejercicios/series se hereda de la rutina seleccionada
- El historial de pesos usados proviene de sesiones completadas previas

Esta dependencia ya está satisfecha según el PRD de rutinas (001-gestion-rutinas.md) y la implementación existente en `src/modules/routines/`.

### Nota sobre implementación del commit masivo

El usuario sugirió investigar las siguientes opciones para el guardado final:

- **Cola de Postgres**: Para procesar la transacción masiva de forma robusta.
- **Procesamiento asíncrono**: Si el volumen de datos es grande, considerar procesar en background.
- **Funcionalidades de Supabase**: Explorar batch operations o RPC para el commit atómico.

Estas son consideraciones técnicas para la fase de arquitectura, no requisitos de producto.
